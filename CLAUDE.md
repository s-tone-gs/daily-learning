# このディレクトリのルール

ここは日々の学習内容をカテゴリ別に整理して保存する場所です。

## 保存の運用

ユーザーから「保存して」「これをファイルに残して」のように依頼されたら、直前のClaudeとの説明・回答内容をそのままMarkdownファイル化して保存する。

1. 内容からカテゴリを判断し、「カテゴリ: ◯◯ でよいですか?」とユーザーに確認する
   - 既存のカテゴリディレクトリがあれば再利用する（`ls`で確認する）
   - 似た内容は同じカテゴリにまとめる
   - 新規カテゴリを作る場合はその旨をユーザーに伝える
2. ファイル名は `YYYY-MM-DD-トピック名.md`とし、日付は実行日を使う
3. ファイル冒頭に以下のメタ情報を記載する

```markdown
# トピック名

- 学習日: YYYY-MM-DD
- カテゴリ: ◯◯

---
```

4. 本文には直前の説明・回答内容をそのまま転記する（要約や書き換えはしない）
   - このファイルは生のQ&Aログであり、暫定的なものとして扱う

## 体系化の運用

学習は「点を得る」だけでは終わらせない。**1つのテーマの学習が一区切りついたタイミングで、都度Claudeから体系化を提案する**(ユーザーからの指示を待たない)。

1. それまでの生のQ&Aログ(複数ファイルにまたがる場合も含む)の内容を、他者に説明できるレベルの体系的な資料に再構成することを提案する
   - 単なる要約ではなく、Q&A形式から説明資料としての構成に組み直す
   - 構成は「動機・背景の説明」からではなく、**最も自明・公理的な原理から始め、そこから具体的な仕組み・実装へとブレイクダウンしていく**トップダウン形式にする(ユーザーの好み)
     - 例: 「なぜ必要か」より先に「そもそも満たすべき抽象的な条件・定義は何か」を最初に置き、その条件をどう実現するか → 具体的な技術・規格、という順で展開する
   - 抜けている繋がりがあれば補い、論理の流れが一本通った文書にする
   - **平易な言葉で書く**。難しい言い回しや論文調の硬い表現は使わない。後でユーザー自身が読み返したときに分からなくなる文章は無意味なので避ける(ユーザーの好み)
     - 注意: 「平易にする」は言葉遣いの話であり、**重要な前提情報(誰が提唱したか、historical/原理的な裏付け、なぜ大事なのかという位置づけ)を削ることではない**。言い回しを簡単にしつつ、内容の重みは保持する
     - 平易さと厳密さのバランスは、その資料の目的・読み手によって変わる。判断に迷ったら省略せず、ユーザーに相談する
2. ユーザーの同意が得られたら、体系化資料を作成する
   - ファイル名は `トピック名.md`(日付なし)とし、カテゴリディレクトリ直下に置く。後から追記・更新していく「育てるドキュメント」として扱う
   - 同じトピックの体系化資料が既にあれば、新しい学習内容を統合して更新する
3. 体系化資料の作成後、元になった生のQ&Aログファイル(`YYYY-MM-DD-トピック名.md`)は削除し、体系化資料に一本化する
4. 体系化資料の更新後、今後その内容を学ぶ人が参照する「学びの資料」になることを意識し、過不足のない説明になっているか確認する

## 議論・修正の姿勢

ユーザーから質問や指摘を受けたとき、それだけで「自分が間違っていた」と即断しない。まず自分で内容を再検証し、正しいと判断できるなら、その根拠を毅然と示して主張する。誤りがある場合は直すが、その際も「なぜ誤りだったか」を説明する。指摘を受けて文書を修正するときは、元の主張が部分的に正しい場合、それを否定形で書き換えるのではなく、新しい視点を前提条件・補足として正しく位置づける。
