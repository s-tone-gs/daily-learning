# 情報セキュリティの基本

- カテゴリ: セキュリティ
- 最終更新: 2026-06-20

---

## 当たり前の要求

情報をやり取りするとき、当たり前に求めることが3つある。

- 見るべき人にだけ見てほしい(それ以外の人には見られたくない)
- 受け取った情報の内容が正しくあってほしい(誰かに書き換えられていない)
- 情報がちゃんと届いてほしい(必要なときに受け取れる)

これは当たり前すぎて、特に説明はいらないレベルの話。問題は、この当たり前を「情報通信」という仕組みの上でどうやって実現するか、ということ。

これを情報セキュリティの世界では「機密性・完全性・可用性」と呼ぶ(英語でConfidentiality, Integrity, Availability。略してCIA)。

| 当たり前の要求 | 呼び方 |
|---|---|
| 見るべき人にだけ見てほしい | 機密性 |
| 内容が正しくあってほしい | 完全性 |
| 情報が届いてほしい | 可用性 |

## 実現方法を考えるには、情報通信の特徴を知る必要がある

当たり前の要求が分かっただけでは、どうやって実現すればいいかは分からない。実現方法は、情報通信という仕組みがどんな特徴を持っているかによって決まる。

情報通信には、こんな特徴がある。

- 電波やネットワークを使うので、途中の情報に誰でも触れられる可能性がある(占有できない)
- データはコピーしても劣化しない。コピーと元のデータを見分けられない
- データを書き換えても、書き換えた跡が残らない
- データは0と1の組み合わせなので、数式で処理・変換できる
- 通信は機械の速さで行われるので、妨害や攻撃も機械の速さで起きる

「当たり前の要求」と「情報通信の特徴」の両方が分かって、初めて「なぜこの方法で実現しているのか」が説明できる。

## 機密性:見るべき人にだけ見てほしい → 暗号化

通信は誰でも触れられる可能性がある(占有できない)。だから「触れられないようにする」ことはできない。できるのは「触れられても意味が分からない状態にする」こと。これが暗号化。

データは数式で変換できるという特徴を使って、決まった「鍵」を持つ人だけが元の内容に戻せるようにする。鍵を知らない人には、暗号化されたデータはただの意味のない文字列にしか見えない。

### 鍵をどう安全に渡すか

暗号化するには、事前に鍵を共有しておく必要がある。しかし、その鍵を渡す瞬間自体がまだ暗号化されていないので、そこを盗み見られたら鍵が漏れてしまう。

これを解決するのが「鍵交換」という仕組み(Diffie-Hellmanという方法など)。数学のトリックを使って、鍵そのものを一度も送らずに、お互いが同じ鍵を計算できるようにする。途中のやり取りを盗み見られても、そこから鍵自体を計算することは(現実的な時間では)できない。

### なぜ「鍵」という方法が選ばれたのか

昔は「暗号化のやり方(アルゴリズム)自体を秘密にする」方法も使われていた(シーザー暗号、エニグマなど)。しかしこの方法には弱点がある。

- やり方が一度解析されると、その仕組み全体が使えなくなる
- やり方自体を秘密にしているので、本当に安全かどうか他の人が検証できない
- 知らない相手とも安全に通信したい場合、やり方を安全に伝える手段が別に必要になる

そこで現代の暗号は、「やり方(アルゴリズム)は公開してよい。秘密にするのは"鍵"だけにする」という考え方に変わった。これを「ケルクホフスの原理」と呼ぶ。19世紀の暗号学者ケルクホフス(Kerckhoffs)が提唱したもので、現代暗号の大前提になっている考え方。(参考: Claude Shannonによる言い換え「敵はシステムを知っている」)

これにより、

- やり方を公開できるので、専門家がレビューして本当に安全かを検証できる
- 同じやり方をみんなが使えるので、メーカーが違う機器同士でも通信できる
- 秘密にする部分が鍵だけなので、漏れたら鍵を変えるだけで済む(やり方自体を作り直す必要がない)

という利点が生まれる。

### 鍵を使い捨てにする理由

接続するたびに新しい鍵を作る設計を「Forward Secrecy(前方秘匿性)」と呼ぶ。1つの鍵が将来漏れても、他のやり取りには影響しないようにするための工夫。これにより、何かが漏れたときの被害を、その1回のやり取りだけに限定できる。

## 完全性:内容が正しくあってほしい → ハッシュ・署名

データは書き換えても跡が残らないし、コピーしても元と見分けがつかない。だから「書き換えられていないこと」を確認する仕組みを、数式を使って人工的に作る必要がある。

これが「ハッシュ関数」や「デジタル署名」。元のデータから特定の計算をして「指紋」のような短い値を作る。内容が少しでも変わると、その指紋も全く別の値になる。受け取った人は、自分で計算した指紋と、送られてきた指紋を比べることで「書き換えられていないか」を確認できる。

## 可用性:情報が届いてほしい → 仕組みでの冗長化

通信は機械の速さで動くので、妨害も機械の速さで起きる(大量のアクセスを送り込んで止める、など)。1つの経路や1つのサーバーだけに頼っていると、簡単に止められてしまう。

だから、複数の経路・複数のサーバーを用意しておき、1つが止まっても他で受け取れるようにする。これを「冗長化」と呼ぶ。

### 可用性は他の2つと少し違う性質を持つ

機密性・完全性は「読めるべきでない人が読めたか」「内容が変わっていたか」という、時代に関係なく決まる基準。

可用性は違う。「見たい時に見られる」の"見たい時"は、その時代の技術でどこまで速くできるかに左右される基準。技術が「ほぼ即時にアクセスできる」を実現できるようになった結果、それが当たり前の基準になり、そこからの遅れや不通が初めて「問題」として意識されるようになった。技術が進むほど、可用性に求められる基準も上がっていく。

## まとめ

```
当たり前の要求: 見るべき人に見てほしい / 内容が正しくあってほしい / 情報が届いてほしい
  +
情報通信の特徴: 占有できない / コピーされても見分けがつかない / 書き換えの跡が残らない / 数式で扱える / 機械の速さで動く
  ↓
実現方法: 暗号化(+鍵交換、ケルクホフスの原理、Forward Secrecy) / ハッシュ・署名 / 冗長化
```
