# Wi-Fiセキュリティと暗号化の基礎

- カテゴリ: セキュリティ
- 最終更新: 2026-06-20

---

## 1. 暗号化が満たすべき公理

暗号化を最も抽象化すると、次の一文に要約できる。

> **第三者が解読できず、意図した受信者だけが元の内容に復号できる変換であること**

ここから先のすべての仕組み(鍵、鍵交換、規格、リスク)は、この一文をどう具体的に実現するか、というブレイクダウンに過ぎない。

## 2. 公理を満たす抽象構造:共有の秘密(鍵)

上記の公理を満たす最も基本的な構造は、次の形である。

> 送信者と受信者だけが知っている「共有の秘密(鍵)」を使ってデータを変換(暗号化)し、同じ鍵を使って元に戻す(復号)。鍵を知らない第三者は元の内容を復元できない。

```
平文 --(共有の鍵で暗号化)--> 暗号文 --(同じ鍵で復号)--> 平文
```

これが**対称鍵暗号**の基本構造であり、Wi-Fiの暗号化(AES-CCMPなど)もこの構造に基づく。

### 公理を満たす方法は鍵以外にもあり得た

公理(第三者が解読できず、受信者のみ復号できる)を満たす手段は、歴史的に鍵方式以外にも存在した。

| 方式 | 概要 | 採用されなかった/限定的な理由 |
|---|---|---|
| アルゴリズム自体を秘密にする | 変換ルールそのものを秘密にする(シーザー暗号、エニグマなど) | 一度方式が解析されるとシステム全体が無効化される。第三者による強度の検証ができない。不特定多数との相互運用が困難 |
| ステガノグラフィ | 通信していること自体を隠す(画像にデータを隠すなど) | 見つかれば無防備。「解読不能化」ではなく「隠蔽」という別の問題を解く手法 |
| 物理的な秘密経路 | 信頼できる人が直接手渡し、専用回線など | デジタルで不特定多数とリアルタイム通信する用途には非現実的 |

### なぜ「鍵」が選ばれたのか:ケルクホフスの原理

19世紀の暗号学者ケルクホフス(Kerckhoffs)が提唱し、現代暗号の大前提になっている原理がある。

> 暗号方式(アルゴリズム)は攻撃者に知られていると仮定しても安全であるべきで、秘密にすべきはただ一つ、「鍵」だけにする

(Claude Shannonによる言い換え: 「敵はシステムを知っている」)

この原理により、以下が同時に実現できる。

- **検証可能性**: アルゴリズムを公開できるため、専門家がレビューし数学的に強度を検証できる(AESは公開コンテストで選定された)
- **標準化・相互運用性**: 同じ公開アルゴリズムを実装すれば、メーカーが違っても通信できる
- **管理コストの低さ**: 秘密にすべき部分が小さなデータ(鍵)だけなので、漏れたら鍵だけ変更すればよく、アルゴリズム自体を作り直す必要がない
- **個別化**: 同じアルゴリズムでも鍵が違えば結果(暗号文)は別物になる

つまり「鍵」は、**「公開・検証可能な変換手順」と「小さく交換・更新しやすい秘密」を分離するための、公理を満たす最も実用的な抽象化**として採用された。

## 3. 派生する問題:秘密(鍵)をどう安全に確立するか

「共有の秘密を持つ」という構造自体が、新しい問題を生む。

- 暗号化通信をするには事前に秘密を共有する必要がある
- しかし、その秘密を共有する瞬間自体はまだ暗号化されていない(平文)
- その瞬間を盗聴されれば、秘密自体が漏れてしまう

これを解決するのが**鍵交換アルゴリズム**(Diffie-Hellmanなど)である。数学的な性質を利用し、秘密そのものを一度も送らずに、両者が同じ秘密の値を計算できるようにする仕組みである。

| 鍵交換方式 | 使われる場面 |
|---|---|
| 4-way handshake | WPA2。パスワード(PSK)を材料に、接続ごとに使い捨ての鍵を生成 |
| SAEハンドシェイク | WPA3-Personal。パスワードを材料にしつつ、より安全な手法で鍵を生成 |
| OWE(Enhanced Open) | WPA3。パスワードなしで、匿名のDiffie-Hellman交換のみで鍵を生成 |

### 派生する設計判断:鍵を使い捨てにする(Forward Secrecy)

接続ごとに新しい鍵を生成する設計判断には、**Forward Secrecy(前方秘匿性)** という思想がある。

> 1つの鍵が将来漏れても、過去・未来の他のセッションの通信には影響しないようにする

```
アルゴリズムを公開する(ケルクホフスの原理)
  → 第三者が検証でき、強度を保証できる
  → 秘密にすべき部分を「鍵」という小さい単位に分離できる
  → 鍵は機器のやり取りごとに使い捨てで生成できる(Forward Secrecy)
  → 1つの鍵が漏れても、被害範囲はその1セッションのみに限定される
```

### パスワードの有無と暗号化の有無は別の軸

「パスワードがあるから暗号化される」は不正確な因果関係である。暗号化を直接決めているのは**鍵交換の手順が動いているかどうか**であり、パスワードはその手順への入力材料の一つに過ぎない。

| 認証方式 | パスワード | 暗号化 |
|---|---|---|
| Open(オープン) | なし | されない(鍵交換の手順自体が存在しない) |
| WPA2/WPA3-Personal | あり | される(パスワードを材料に鍵を生成) |
| WPA-Enterprise | なし(証明書等) | される(EAP認証の結果から鍵を生成) |
| WPA3 Enhanced Open(OWE) | なし | される(匿名の鍵交換のみ) |

OWEのように「パスワードなしでも暗号化される」方式が存在することが、両者が別概念であることを示す。実務上「パスワードなし=暗号化なし」と言われがちなのは、現状ほとんどのパスワードなしWi-FiがOpen方式(鍵交換なし)で運用されているという**普及状況による相関**であり、論理的必然ではない。

## 4. 具体的な実装:Wi-Fiにおける規格と機器

ここまでの抽象的な構造(鍵、鍵交換、ケルクホフスの原理、Forward Secrecy)を、Wi-Fiという具体的な通信手段の上に実装したものが、以下のAPと規格である。

### アクセスポイント(AP)

無線LANの電波を発信し、デバイスをネットワークに接続させる中継機器。

```
[スマホ] ⇄ (電波) ⇄ [アクセスポイント] ⇄ (有線LAN/光回線など) ⇄ [インターネット]
```

| 用語 | 役割 |
|---|---|
| アクセスポイント(AP) | 無線(Wi-Fi)接続の出入口を提供する |
| ルーター | 異なるネットワーク間の通信を中継・振り分けする |

家庭用Wi-Fiルーターは両方を1台に統合した機器。スマホのテザリングも、スマホ自身が簡易的なAPになっている状態。

### WPA規格

Wi-Fi Allianceが定める、「Wi-Fi通信をどう暗号化・認証するか」の業界規格。

| 規格 | 暗号化方式 | 特徴 |
|---|---|---|
| WPA | TKIP | WEPの欠陥を緊急に補う暫定規格(2003年) |
| WPA2 | AES(CCMP) | 長く標準として使われた(2004年) |
| WPA3 | AES + SAE | 総当たり攻撃への耐性を強化した現行規格(2018年) |

認証モードには家庭用の**WPA-Personal(PSK、共通パスワード)**と、企業用の**WPA-Enterprise(個別認証)**がある。

## 5. 公理の前提が崩れるケース:電波の傍受とフリーWi-Fiのリスク

「第三者が解読できない」という公理は鍵によって満たされるが、これはあくまで「データの内容を読めない」ことを保証するだけである。前提として、無線LANは電波という性質上、**傍受されること自体は誰にも防げない**。

- 電波の届く範囲にいれば、モニターモード対応の機器があれば誰でもキャプチャできる(APに接続している必要はない)
- 暗号化は「傍受されたデータが読めない」状態を作るためのものであり、「傍受されないようにする」仕組みではない

さらに、暗号化の公理は「鍵を共有した相手以外は読めない」ことしか保証しないため、**鍵を共有する相手(AP)自体が信頼できるか**という別軸の問題が残る。

| パターン | 内容 |
|---|---|
| 暗号化なし(オープンWi-Fi) | 通信内容がそのまま盗聴される |
| Evil Twin(偽のアクセスポイント) | 攻撃者が本物と同じ(または似た)SSID名で偽APを設置し、利用者を誤って接続させる。攻撃者自身が鍵交換の相手になるため、暗号化されていても攻撃者は内容を復号できる(中間者攻撃) |
| 本物のAPだがパスワードが共有されている(店舗のWPA2-Personal) | 店内の全員が同じパスワードを知っているため、パスワードを知る別の客が、自分以外の客の鍵交換(4-way handshake)をキャプチャできれば、原理上その客の通信を復号できる可能性がある |

テザリング(自分のスマホ)とフリーWi-Fiの違いは、この「鍵交換の相手(AP)の信頼性」がどう変わるかという観点で整理できる。

| 観点 | テザリング(自分のスマホ) | フリーWi-Fi |
|---|---|---|
| APの信頼性 | 高い(自分の機器) | 不明(Evil Twinの可能性) |
| パスワードの共有範囲 | 自分のみ | 不特定多数が知っている場合がある |
| 暗号化の有無 | 通常あり | ない場合もある |

「暗号化されているから安全」と短絡的に考えるのではなく、「誰がそのAPを管理し、パスワードを誰が知っているか」というAP自体の信頼性も合わせて見る必要がある。
